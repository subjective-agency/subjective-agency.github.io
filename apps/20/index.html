    ∞ ARR-20: PDFeX

    General

    Today’s software landscape offers a large number of ways to process PDF files. For the purposes of retrieving as clean content as possible, following ways were considered:

    • Conventional tools: StirlingPDF, borb
    • AI (JIna)
    • OCR (easyOCR)

    Metadata on PDF files is not guaranteed to contain useful info, but often does.

    StirlingPDF and borb

    StirlingPDF is a self-hosted PDF-multitool that works as a webservice. In the context of the app, actions like removing blank pages, splitting by chapters, and extracting raw text are relevant. If used, instance of StirlingPDF must be launched alongside the application as part of the docker compose setup.

    borb is a Python library that can be used for all kinds of things, but in the context of the app, only retrieving metadata is relevant, since everything else is easier done with Stirling.

    JinaAI

    Jina is an AI company that maintains a number of endpoints for AI-based content parsing, segmentation, etc. One endpoint relevant in the context of the app is r.jina.ai, which retrieves content of any web page, including web-hosted PDFs and DOES NOT require API key.

    However, there is no telling how long this service would be available for. Also: would only work on PDFs hosted on public web-pages.

    OCR

    Both Stirling-based and Jina-based ways both would produce raw text as it was recognized by the original creators of the document, that is, are guaranteed to contain numerous unwanted artifacts that require cleaning out.

    Converting PDFs into images and re-doing OCR is likely to produce a better quality-result, because modern OCR tools (like easyOCR ) are using AI models to enhance the result.

    Downside is that OCR is a significantly more resource-demanding process, and more prone to breaking, too.
